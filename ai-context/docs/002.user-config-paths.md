# 用户配置目录与 home 目录配置文件说明

> 本文回答：**`claw` 是否像 Claude Code 那样在用户家目录有一个 `~/.claude/settings.json`？**
>
> 短回答：**没有，但有自己的等价物 `~/.claw/settings.json`。** 路径名、解析顺序、与 Claude Code 的关系下面分章节展开。所有结论均来自代码，不是 README/USAGE 的承诺。

## 直接对比

| 工具 | 用户级配置文件 | 项目级配置文件 |
|---|---|---|
| **Claude Code** | `~/.claude/settings.json`（以及 `.claude/`、`.claude.json` 等） | `<repo>/.claude/settings.json` 等 |
| **`claw` (本项目)** | `~/.claw/settings.json`（外加 legacy `~/.claw.json`） | `<repo>/.claw.json`、`<repo>/.claw/settings.json`、`<repo>/.claw/settings.local.json` |

也就是说：**`claw` 把 Claude Code 的 `.claude` 目录命名整体改写成了 `.claw`**。文件结构和分层语义保持类似，但目录名和文件名都不一样。

> ⚠️ 两条容易踩的常识：
>
> 1. 仓库根目录下确实存在一个 `.claude.json`（见 `@d:\codingnet\claw-code\.claude.json`），但那是**仓库自身**给上游 Claude Code 客户端预留的兼容文件，**不是** `claw` 自己运行时会读的配置。
> 2. `claw` **不会**读 `~/.claude/settings.json` 当 runtime 配置。哪怕你已经用 Claude Code 多年，`claw` 也不会自动复用那份 `settings.json`。

## `claw` 实际识别的所有配置文件（按合并顺序）

代码路径：`crates/runtime/src/config.rs::ConfigLoader::discover`

加载顺序如下，**later 覆盖 earlier**（见 `ConfigLoader::load` 中的 `deep_merge_objects`）：

| # | 路径 | 来源标签 | 用途 |
|---|---|---|---|
| 1 | `~/.claw.json` | `User`（legacy） | 单文件用户配置（旧风格） |
| 2 | `~/.claw/settings.json` | `User` | **正式的用户级 settings**（Claude Code `~/.claude/settings.json` 的等价物） |
| 3 | `<repo>/.claw.json` | `Project`（legacy） | 单文件项目配置（旧风格） |
| 4 | `<repo>/.claw/settings.json` | `Project` | 项目级 settings |
| 5 | `<repo>/.claw/settings.local.json` | `Local` | 机器本地覆写（已被 `.gitignore` 排除） |

合并示意：

```text
~/.claw.json
   └─→ ~/.claw/settings.json
          └─→ <repo>/.claw.json
                 └─→ <repo>/.claw/settings.json
                        └─→ <repo>/.claw/settings.local.json   ← 最高优先级
```

每个文件都只接受同一组键（`model`、`aliases`、`permissions`、`permission_rules`、`hooks`、`plugins`、`mcp`、`oauth`、`provider_fallbacks`、`trusted_roots`、`sandbox`），**不接受 `base_url` / `api_key` / `auth_token` 等凭据字段**（这部分参见 `@d:\codingnet\claw-code\ai-context\docs\001.third-party-api-config.md`）。

## `~/.claw/` 是怎么解析出来的（重要：Windows 有坑）

代码：`crates/runtime/src/config.rs:558-565`

```rust
pub fn default_config_home() -> PathBuf {
    std::env::var_os("CLAW_CONFIG_HOME")
        .map(PathBuf::from)
        .or_else(|| std::env::var_os("HOME").map(|home| PathBuf::from(home).join(".claw")))
        .unwrap_or_else(|| PathBuf::from(".claw"))
}
```

解析顺序：

1. 如果设置了 `CLAW_CONFIG_HOME` → 直接用它（**不会**自动追加 `.claw`，你给什么就是什么）。
2. 否则，如果设置了 `HOME` → 用 `$HOME/.claw`。
3. 否则，**回退到相对路径** `./.claw`（即当前工作目录下的 `.claw/`）。

### Windows 上的真实陷阱

PowerShell **默认不设 `HOME`**，只设 `USERPROFILE`。这意味着：

- 如果你**没有**显式设过 `HOME` 或 `CLAW_CONFIG_HOME`，`default_config_home()` 会落到 fallback 分支 `./.claw`。
- 在仓库目录里运行时，**`~/.claw/settings.json` 实际上指的是 `<repo>/.claw/settings.json`**（与项目级路径重合）。
- 如果你把 user-level 配置写在了 `C:\Users\<you>\.claw\settings.json`，但没设 `HOME`，**`claw` 不会去读它**。

> Git for Windows / WSL / 部分 IDE 启动子进程时会自动注入 `HOME`，所以这个问题在不同终端表现不一致。最稳妥的做法是**显式设置** `HOME` 或 `CLAW_CONFIG_HOME`。

### Windows 上的修复方案

任选一种持久化方式（参考 `@d:\codingnet\claw-code\ai-context\docs\001.third-party-api-config.md` 的轴二）：

```powershell
# 方案 1：把 USERPROFILE 映射到 HOME（最贴近 Linux 习惯）
[Environment]::SetEnvironmentVariable("HOME", $env:USERPROFILE, "User")

# 方案 2：直接指定 CLAW_CONFIG_HOME（更明确，不影响其他工具）
[Environment]::SetEnvironmentVariable("CLAW_CONFIG_HOME", "$env:USERPROFILE\.claw", "User")
```

设置后**新开终端**生效。验证方法：

```powershell
.\rust\target\debug\claw.exe doctor
```

`doctor` 输出会显示 `Config home` 实际指向的目录。

## `.claude/` 在 `claw` 里到底起什么作用？

通读 `crates/runtime/src/config.rs`、`crates/tools/src/lib.rs`、`crates/commands/src/lib.rs` 后，`.claude/` 的命中点**只有三处**，都不是 runtime 配置：

| 用途 | 项目级路径 | 用户级路径 | 代码位置 |
|---|---|---|---|
| Skills 查找 | `<repo>/.claude/skills/` | `~/.claude/skills/`、`~/.claude/skills/omc-learned/` | `tools/src/lib.rs::push_*_skill_lookup_roots` |
| Agents 查找 | `<repo>/.claude/agents/` | `~/.claude/agents/` | `commands/src/lib.rs` |
| 旧版 commands 兼容 | `<repo>/.claude/commands/` | `~/.claude/commands/` | 同上（标记为 `LegacyCommandsDir`） |

也就是说：**`claw` 主动复用了 Claude Code 用户已经积累的 skill/agent/command 资产**，但**不复用** `settings.json`、`CLAUDE.md`、permission 规则等 runtime 决策。它和 Claude Code 共享内容资产，但保留独立的运行时身份。

完整查找根目录列表（来自 `claw skills help`，源代码在 `commands/src/lib.rs:3795-3829`）：

```text
项目级：.claw/skills, .omc/skills, .agents/skills, .codex/skills, .claude/skills
用户级：~/.claw/skills, ~/.omc/skills, ~/.codex/skills, ~/.claude/skills, ~/.claude/skills/omc-learned
旧版兼容：legacy /commands
```

注意里面**没有** `.claude/settings.json` 这一项 —— 再次说明该文件不会被 `claw` runtime 读到。

### 跨工具的环境变量

`crates/tools/src/lib.rs:3254-3262` 在做 skill 查找时同时识别：

- `CLAW_CONFIG_HOME` —— `claw` 自己的配置根
- `CODEX_HOME` —— OmX (`oh-my-codex`) 的配置根
- `HOME` 或 `USERPROFILE` —— 用户家目录（**这里**两者都会被看，是和 `default_config_home` 不一样的地方）
- `CLAUDE_CONFIG_DIR` —— 仅出现在 skill 测试里，用于覆盖 `~/.claude/skills/omc-learned` 的位置

> 这个不一致是个潜在的 polish 点：runtime 的 `default_config_home` 应该也回退到 `USERPROFILE`，但目前没有。**用户层面的稳妥做法仍是显式设 `HOME` 或 `CLAW_CONFIG_HOME`。**

## OAuth 凭据文件（独立于 settings）

代码：`crates/runtime/src/oauth.rs::credentials_path`

```text
<config_home>/credentials.json
```

`config_home` 同上规则解析。例如在 Linux/macOS 上是 `~/.claw/credentials.json`，Windows 上若设了 `HOME` 则是 `%HOME%\.claw\credentials.json`。

> 这是**遗留文件**：`claw login` 已被移除，`claw doctor` 把这里的 OAuth token 显式标为 `legacy saved OAuth credentials are present but unsupported`。新部署不要依赖它，请改用 `ANTHROPIC_API_KEY` 或 `ANTHROPIC_AUTH_TOKEN` 环境变量。

## Sessions / Worker state（不是 settings，但住在同一目录树）

虽然不是配置文件，顺带列出 `~/.claw/`（或 `<config_home>/`）下其他可能出现的内容，方便排查：

| 路径 | 写入者 | 用途 |
|---|---|---|
| `<config_home>/credentials.json` | 旧版 OAuth flow（已弃用） | OAuth token 缓存 |
| `<repo>/.claw/sessions/<id>.jsonl` | REPL / one-shot prompt | 单次会话回放（被 `.gitignore` 排除） |
| `<repo>/.claw/worker-state.json` | Worker 启动 | 最近一次 worker 元数据，被 `claw state` 读取 |
| `<repo>/.claw/settings.local.json` | 用户手写 | 本机覆写（被 `.gitignore` 排除） |

注意区分：**项目级 `.claw/`** 与 **用户级 `~/.claw/`** 是两个不同位置；session 文件只放在项目级，settings.json 两边都可能存在。

## 一个最小工作示例（PowerShell, Windows）

假设你想要：
- 用户级默认走 OpenRouter 的 `gpt-4.1-mini`
- 把这一选择写在用户级 settings 里，新仓库不用重复配置
- 在 `claw` 看得到 `~/.claw/settings.json` 而不是 `./.claw/settings.json`

步骤：

```powershell
# 1. 让 claw 能找到 home
[Environment]::SetEnvironmentVariable("CLAW_CONFIG_HOME", "$env:USERPROFILE\.claw", "User")

# 关闭并重开 PowerShell，让用户级 env 生效

# 2. 创建用户级配置目录与 settings.json
New-Item -ItemType Directory -Force "$env:USERPROFILE\.claw" | Out-Null
@'
{
  "model": "openai/gpt-4.1-mini",
  "aliases": {
    "fast": "claude-haiku-4-5-20251213"
  }
}
'@ | Set-Content -Encoding UTF8 "$env:USERPROFILE\.claw\settings.json"

# 3. 配置 OpenRouter 凭据（这部分必须走环境变量）
[Environment]::SetEnvironmentVariable("OPENAI_BASE_URL", "https://openrouter.ai/api/v1", "User")
[Environment]::SetEnvironmentVariable("OPENAI_API_KEY",  "sk-or-v1-...",                "User")

# 关闭并重开 PowerShell

# 4. 验证
.\rust\target\debug\claw.exe doctor
.\rust\target\debug\claw.exe prompt "say hello"
```

`doctor` 中会列出加载的配置入口，确认 `~/.claw/settings.json` 已被识别。

## 总结

- **没有** `~/.claude/settings.json` 这个概念在 `claw` 里。
- **有** `~/.claw/settings.json`，是 Claude Code 同名文件的等价物，遵循 `.claw` 命名空间。
- 五层配置文件（`~/.claw.json` → `~/.claw/settings.json` → `<repo>/.claw.json` → `<repo>/.claw/settings.json` → `<repo>/.claw/settings.local.json`）按 later-wins 顺序合并。
- `~/.claude/` 只在查找 **skills / agents / 旧版 commands** 时被扫描，与运行时设置无关。
- Windows 上务必显式设 `HOME` 或 `CLAW_CONFIG_HOME`，否则 `~/.claw/settings.json` 实际会落到当前工作目录下的 `.claw/`，行为难以预期。
- API Key、Base URL 等凭据**不**在 settings 里，请走环境变量（详见 `@d:\codingnet\claw-code\ai-context\docs\001.third-party-api-config.md`）。
