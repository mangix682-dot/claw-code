# 第三方 API 地址与 API Key 配置方法

> 本文基于代码（不是 README/USAGE 的承诺）整理，回答 **"`claw` 究竟通过哪些渠道读取 API 端点和密钥"** 这一问题。所有结论都附带源文件锚点，便于后续验证。

## 总览：仅有 1 个真实生效的来源 —— 环境变量

通过通读 `crates/api/src/providers/` 与 `crates/runtime/src/config.rs`，可以确认：

- **API Key 与 Bearer Token** 只从环境变量读取。
- **Base URL（API 地址）** 只从环境变量读取。
- **CLI flag 不支持**：`crates/rusty-claude-cli/src/main.rs` 解析的 flag 只有 `--model`、`--output-format`、`--permission-mode`、`--base-commit`、`--reasoning-effort`、`--resume`、`--allowedTools`、`--session`、`--output`、`--dangerously-skip-permissions` 等。**没有 `--api-key` 或 `--base-url`**。
- **配置文件不支持**：`.claw.json` / `.claw/settings.json` / `.claw/settings.local.json` 解析时只识别 `model`、`aliases`、`permissions`、`permission_rules`、`hooks`、`plugins`、`mcp`、`oauth`、`provider_fallbacks`、`trusted_roots`、`sandbox` 等键。**没有 `base_url` 或 `api_key` 字段**。
- **旧版 `claw login`**：已被移除。再次执行只会得到 `\`claw login\` has been removed. Set ANTHROPIC_API_KEY or ANTHROPIC_AUTH_TOKEN instead.`（见 `crates/rusty-claude-cli/src/main.rs` 中 `removed_auth_surface_error`）。
- **遗留 OAuth 凭据文件** `~/.claw/credentials.json`：代码会读，但 `claw doctor` 把它标记为 `legacy saved OAuth credentials are present but unsupported`，必须以环境变量为准。**不要把它当作主要配置渠道**。

也就是说：**"配置第三方 API 地址和 Key" 在 `claw` 中本质上 = 设置正确的环境变量**。区别仅在于：
1. 你要走哪一组环境变量（取决于目标 provider）。
2. 你用什么方式设置这些环境变量（持久化范围）。

下面分别从这两个轴展开。

---

## 轴一：按 Provider 选择正确的环境变量组合

| Provider 类型 | API Key 环境变量 | Base URL 环境变量 | 默认 Base URL | 来源文件 |
|---|---|---|---|---|
| Anthropic（直连或 Anthropic-兼容代理） | `ANTHROPIC_API_KEY` *或* `ANTHROPIC_AUTH_TOKEN` | `ANTHROPIC_BASE_URL` | `https://api.anthropic.com` | `crates/api/src/providers/anthropic.rs` |
| OpenAI / OpenRouter / Ollama / 自建 OpenAI-兼容服务 | `OPENAI_API_KEY` | `OPENAI_BASE_URL` | `https://api.openai.com/v1` | `crates/api/src/providers/openai_compat.rs` |
| xAI（Grok 系列） | `XAI_API_KEY` | `XAI_BASE_URL` | `https://api.x.ai/v1` | 同上 |
| Alibaba DashScope（Qwen、Kimi） | `DASHSCOPE_API_KEY` | `DASHSCOPE_BASE_URL` | `https://dashscope.aliyuncs.com/compatible-mode/v1` | 同上 |

### 重要规则：API Key 与 Bearer Token **不可互换**

代码 `AuthSource::from_env`（`crates/api/src/providers/anthropic.rs`）严格区分两类凭据：

- `ANTHROPIC_API_KEY` → 走 HTTP 头 `x-api-key: sk-ant-...`，适合 `sk-ant-*` 形式的密钥。
- `ANTHROPIC_AUTH_TOKEN` → 走 HTTP 头 `Authorization: Bearer ...`，适合 OAuth access token 或 Anthropic-兼容代理签发的 bearer。

把 `sk-ant-*` 误填进 `ANTHROPIC_AUTH_TOKEN` 会得到 `401 Invalid bearer token`。`claw` 已经检测了这个特定错位并在错误消息中追加了修复提示。

### 重要规则：模型名前缀路由会**覆盖** ambient 环境变量

`metadata_for_model`（`crates/api/src/providers/mod.rs`）按模型名首段强制 routing：

- `claude*` → Anthropic（读 `ANTHROPIC_*`）
- `grok*` → xAI（读 `XAI_*`）
- `qwen/*` 或 `qwen-*` → DashScope（读 `DASHSCOPE_*`）
- `kimi/*` 或 `kimi-*` → DashScope（读 `DASHSCOPE_*`）
- `openai/*` 或 `gpt-*` → OpenAI-compat（读 `OPENAI_*`）
- 其他名字 → 由 `detect_provider_kind` 按顺序探测：`OPENAI_BASE_URL+OPENAI_API_KEY` → `ANTHROPIC_*` → `OPENAI_API_KEY` → `XAI_API_KEY` → 默认 Anthropic。

这意味着：
- 即使你同时设了 `ANTHROPIC_API_KEY` 和 `OPENAI_API_KEY`，只要 `--model openai/gpt-4.1-mini`，就会走 OpenAI 而非 Anthropic。
- 反过来，如果你想让本地 Ollama 跑一个名字不带前缀的模型（如 `qwen2.5-coder:7b`），**必须**显式设 `OPENAI_BASE_URL`，否则会被误判为 Anthropic。

### 典型组合示例（PowerShell）

```powershell
# 方案 1：Anthropic-兼容代理（如本地 mock 服务、企业网关）
$env:ANTHROPIC_BASE_URL = "http://127.0.0.1:8080"
$env:ANTHROPIC_AUTH_TOKEN = "local-dev-token"
.\claw.exe --model "claude-sonnet-4-6" prompt "reply with the word ready"

# 方案 2：OpenAI-兼容服务（OpenRouter、自建 vLLM、LM Studio…）
$env:OPENAI_BASE_URL = "https://openrouter.ai/api/v1"
$env:OPENAI_API_KEY  = "sk-or-v1-..."
.\claw.exe --model "openai/gpt-4.1-mini" prompt "..."

# 方案 3：Ollama（本机、通常无需 key）
$env:OPENAI_BASE_URL = "http://127.0.0.1:11434/v1"
Remove-Item Env:\OPENAI_API_KEY -ErrorAction SilentlyContinue
.\claw.exe --model "llama3.2" prompt "..."

# 方案 4：DashScope (Qwen / Kimi)
$env:DASHSCOPE_API_KEY = "sk-..."
.\claw.exe --model "qwen-plus" prompt "..."
.\claw.exe --model "kimi-k2.5"  prompt "..."

# 方案 5：xAI (Grok)
$env:XAI_API_KEY = "..."
.\claw.exe --model "grok-3" prompt "..."
```

### HTTP 代理（独立维度）

`HTTP_PROXY` / `HTTPS_PROXY` / `NO_PROXY`（大小写均可）会被自动识别用于 outbound 请求转发。这与 Base URL **不是同一概念**：
- **Base URL** 改变请求实际打到的 API 端点。
- **HTTP 代理** 是把已经选定的端点请求**隧道**通过代理服务器。

也可在程序内通过 `ProxyConfig::proxy_url`（`crates/api/src/http_client.rs`）作为统一覆写。CLI 用户一般只用环境变量即可。

## 轴二：环境变量的设置方式（持久化范围）

`claw` 自身**不读取 `.env` 文件**，也不解析 `.claw.json` 中的密钥字段。所有方案最终都归结为"让 `claw.exe` 启动时进程环境里包含正确的变量"。下面按 Windows 上常见的持久化范围整理出 5 种实操路径。

### 方法 A：当前 PowerShell 会话（一次性）

```powershell
$env:ANTHROPIC_API_KEY = "sk-ant-..."
$env:ANTHROPIC_BASE_URL = "http://127.0.0.1:8080"
.\rust\target\debug\claw.exe doctor
```

- 范围：仅当前 shell 会话；关闭窗口即失效。
- 适合：临时调试、切换 provider 做对比。
- 不适合：日常使用（每次开新窗口都要重设）。

### 方法 B：PowerShell `$PROFILE` 自动加载

把 export 写进当前用户的 PowerShell 启动脚本：

```powershell
# 编辑 profile（路径示例：C:\Users\<you>\Documents\PowerShell\Microsoft.PowerShell_profile.ps1）
notepad $PROFILE

# 在 profile 末尾追加：
$env:ANTHROPIC_API_KEY = "sk-ant-..."
$env:OPENAI_API_KEY    = "sk-..."
$env:DASHSCOPE_API_KEY = "sk-..."
```

- 范围：当前用户每次启动 PowerShell 时自动加载。
- 适合：日常稳定使用单一组凭据。
- 注意：profile 是明文文件，注意权限与备份策略。

### 方法 C：Windows 用户级环境变量（持久、跨 shell）

```powershell
[Environment]::SetEnvironmentVariable("ANTHROPIC_API_KEY", "sk-ant-...", "User")
[Environment]::SetEnvironmentVariable("OPENAI_BASE_URL",   "https://openrouter.ai/api/v1", "User")
```

或图形界面：`系统属性 → 高级 → 环境变量 → 用户变量`。

- 范围：当前 Windows 用户的所有进程（PowerShell、cmd、IDE 子进程、计划任务等）。
- 设置后**新开**的终端才会看到，已开的终端不刷新。
- 适合：当 `claw` 被 IDE / 编辑器插件 / 后台任务以非交互方式启动时。

### 方法 D：Windows 系统级环境变量（机器范围）

```powershell
# 需要管理员 PowerShell
[Environment]::SetEnvironmentVariable("ANTHROPIC_API_KEY", "sk-ant-...", "Machine")
```

- 范围：所有 Windows 用户、所有进程。
- 适合：服务器部署、多用户共享开发机、CI runner。
- 不适合：个人机器上保存私人 key（其他用户也能读取）。

### 方法 E：单条命令内联注入

PowerShell：

```powershell
& {
    $env:ANTHROPIC_API_KEY = "sk-ant-..."
    $env:ANTHROPIC_BASE_URL = "http://127.0.0.1:8080"
    .\rust\target\debug\claw.exe prompt "hello"
}
```

cmd 一行式：

```cmd
cmd /c "set ANTHROPIC_API_KEY=sk-ant-... && set ANTHROPIC_BASE_URL=http://127.0.0.1:8080 && claw prompt hello"
```

- 范围：仅这一次调用；变量在子进程里存在，回到父 shell 后消失。
- 适合：脚本化、CI 步骤、需要在多套配置间快速切换的演示。

### 方法对比

| 方法 | 持久 | 跨 shell | 跨用户 | 典型场景 |
|---|---|---|---|---|
| A 当前会话 `$env:VAR=...` | ✗ | ✗ | ✗ | 临时调试 |
| B PowerShell `$PROFILE` | ✓（重启 shell 后仍在） | ✗（仅 PowerShell） | ✗ | 日常单用户 |
| C 用户级 SetEnvironmentVariable | ✓ | ✓ | ✗ | IDE 子进程、跨终端 |
| D 系统级 SetEnvironmentVariable | ✓ | ✓ | ✓ | 服务器、CI |
| E 内联 `& { ... }` 或 `cmd /c set ... && ...` | ✗ | ✗ | ✗ | 单次命令、脚本步骤 |

## 与配置文件的关系（一个常见误解）

很多人会问："能不能把 API Key 写进 `.claw.json`？"。答案是**不能**——通读 `crates/runtime/src/config.rs`（`ConfigLoader::discover` + `RuntimeFeatureConfig` 字段集合），配置文件只承载下列内容：

- `model` — 默认模型
- `aliases` — 用户自定义模型别名（如 `"fast": "claude-haiku-4-5-20251213"`）
- `permissions` / `permission_rules` — 权限策略
- `mcp.servers` — MCP server 列表与 transport 配置
- `oauth` — OAuth flow 元数据（不是 API key 本身）
- `provider_fallbacks` — 主 provider 失败时的回退顺序（声明性，不带凭据）
- `hooks` / `plugins` / `sandbox` / `trusted_roots`

也就是说，**配置文件管 "用什么模型 / 怎么 fallback / 工具权限"，环境变量管 "去哪发请求、用什么凭据"**。两者职责分离，不能互相替代。

如果你想让某个工程目录默认走 OpenRouter，**正确的做法是**：

1. `.claw/settings.json` 里设：
   ```json
   { "model": "openai/gpt-4.1-mini" }
   ```
2. 通过方法 B/C 在用户级别设好 `OPENAI_API_KEY` 与 `OPENAI_BASE_URL`。
3. 在该目录下运行 `claw`，模型前缀路由 + ambient env var 会自动指向 OpenRouter。

配置文件解析顺序（later 覆盖 earlier，见 `ConfigLoader::discover`）：
1. `~/.claw.json`
2. `~/.config/claw/settings.json`（在 Windows 上 `~` 由 `dirs::config_dir` 决定，通常映射到 `%APPDATA%\claw\`）
3. `<repo>/.claw.json`
4. `<repo>/.claw/settings.json`
5. `<repo>/.claw/settings.local.json`（已被 `.gitignore` 排除）

## 最终回答："有几种方法"

把"配置第三方 API 地址 + Key"拆成两个问题：

**问题 1：`claw` 从几个**来源**读取这些值？**

- **正式支持的来源：1 个** —— 环境变量。
- 遗留来源：1 个，且被标记为 unsupported —— `~/.claw/credentials.json` 中的旧 OAuth token。
- 不支持：CLI flag、`.claw.json`、`.env` 文件、Windows 凭据管理器。

**问题 2：你可以用几种**方式**让那组环境变量在 `claw` 启动时存在？**

- **5 种实操方式**（详见上文）：
  1. 当前 PowerShell 会话内的 `$env:VAR = "..."`
  2. PowerShell `$PROFILE` 自动加载
  3. `[Environment]::SetEnvironmentVariable(..., "User")`
  4. `[Environment]::SetEnvironmentVariable(..., "Machine")`
  5. 单条命令内联注入（`& { ... }` 或 `cmd /c set ... && ...`）

加上 provider 维度（5 种 provider × API Key/Bearer 两种凭据形态），可以组合出十几种实操配置；但**底层路径只有"环境变量"这一条**。所有"花哨配置"本质都是这一条路径的不同包装。
