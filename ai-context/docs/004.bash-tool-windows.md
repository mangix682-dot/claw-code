# Windows 上 `claw` 的 bash 工具找不到 Python（与其他可执行文件）的根因

> 现象：在 PowerShell 直接 `python xxx.py` 跑得通，但 `claw` 的 REPL 里执行同一条命令报 `✗ bash / program not found`。
>
> 一句话结论：**Python 没事，是 `sh.exe` 没找到**。`claw` 的 bash 工具被硬编码成 `sh -lc <command>`，Windows 上没有 `sh.exe` 时整条命令在还没碰到 `python` 之前就已经挂掉了。

## 用户实际看到的错误

```text
> 帮我写一个快速排序 python 版本放到根目录
... (write_file 成功) ...

╭─ bash ─╮
│  $ python D:\clawtester\quicksort.py
╰────────────╯
✗ bash
program not found

╭─ bash ─╮
│  $ python3 D:\clawtester\quicksort.py 2>nul || py D:\clawtester\quicksort.py
╰────────────╯
✗ bash
program not found
```

两条命令都失败，都是同一条字面消息 `program not found`，没有任何 `sh: python: command not found` 之类的 shell 风格输出。这是关键线索。

## 代码定位（root cause）

`crates/runtime/src/bash.rs` 第 269-270 行（同步路径）和第 296-297 行（async 路径）：

```rust
let mut prepared = Command::new("sh");
prepared.arg("-lc").arg(command).current_dir(cwd);
```

**没有任何 `#[cfg(windows)]` 分支**。无论操作系统是 Linux、macOS、还是 Windows，`claw` 的 bash 工具都是无脑调用 `sh -lc <command>`。

对比一下，同一份代码库里 `crates/runtime/src/hooks.rs:738-742` 是怎么写的：

```rust
fn shell_command(command: &str) -> CommandWithStdin {
    #[cfg(windows)]
    let mut command_builder = {
        let mut command_builder = Command::new("cmd");
        command_builder.arg("/C").arg(command);
        ...
```

`plugins/src/hooks.rs:281-285` 同样写了 Windows 分支。也就是说**hooks 系统知道在 Windows 用 `cmd /C`，但 bash 工具忘了做这件事**。这是上游一个明确的 Windows 可移植性 gap。

## 错误链（为什么是 `program not found` 而不是 `python not found`）

```
1. AI 在工具调用里发出 bash 命令： python D:\clawtester\quicksort.py

2. claw bash 工具：
       Command::new("sh")
           .arg("-lc")
           .arg("python D:\\clawtester\\quicksort.py")
           .spawn()

3. Windows 在 %PATH% 中找 sh.exe → 找不到（默认 Windows 不带 sh）

4. Windows 返回 os error 2 (ERROR_FILE_NOT_FOUND)

5. Rust stdlib 把 io::Error { ErrorKind::NotFound } 渲染成 "program not found"

6. crates/runtime/src/bash.rs 中 `command.output().await?` 把这个错误向上传播

7. claw UI 渲染为：
       ✗ bash
       program not found
```

注意第 5 步的 `program` 指的是 **`sh.exe`**，不是 `python`。如果是 `sh` 跑起来后再找不到 `python`，输出会是 POSIX shell 的标准格式：

```text
sh: python: command not found
exit code 127
```

且 `claw` 会把它当作正常完成的命令输出渲染（红色非空 stderr，但不是 `✗ bash` 这种 spawn 级失败）。

**用户看到的扁平 `program not found` = 启动 sh 这一步就挂了。**

## 为什么 PowerShell 里能跑、`claw` 里不能

| 上下文 | 解析 `python` 的方式 | 是否依赖 `sh.exe` |
|---|---|---|
| PowerShell 终端直接输入 `python xxx.py` | Windows shell 直接查 `%PATH%` 与 App Execution Aliases | ❌ |
| `claw` REPL 中 AI 调用 bash 工具 | `Command::new("sh").arg("-lc").arg("python ...")` | ✅ **必需** |
| `claw` REPL 中 AI 调用 `PowerShell` 工具 | `Command::new("pwsh"\|"powershell")` 直接跑 | ❌ |
| `claw` 的 hooks（钩子脚本） | Windows 下走 `cmd /C` 分支 | ❌ |

也就是说，**bash 工具是 `claw` 整个工具链里少数没做 Windows fallback 的部分**。Python 是否安装、是否在 PATH 都不重要——sh 找不到就根本走不到那一步。

## 如何验证 root cause

在你的 PowerShell 里跑：

```powershell
Get-Command sh -ErrorAction SilentlyContinue
```

如果输出空 / 没有结果 → 确认 `sh.exe` 不在 PATH，与本文分析一致。

进一步在 `claw` REPL 里用一条任意命令验证（哪怕 `pwd` 都行）：

```text
> 在仓库根目录跑一下 pwd
... ✗ bash / program not found
```

只要任何 bash 调用都报 `program not found`，就 100% 是 sh 缺失问题。

## 5 种解法

### 解法 1：装 / 重装 Git for Windows，开启 "Optional Unix tools" 选项（推荐）

下载 <https://git-scm.com/download/win>，安装时在 **"Adjusting your PATH environment"** 步骤选择：

> **Use Git and optional Unix tools from the Command Prompt**

而不是默认的 *Use Git from the Windows Command Prompt*。这一项会把 `C:\Program Files\Git\usr\bin\` 加到 PATH，里面有 `sh.exe`、`bash.exe`、`grep`、`sed`、`awk` 等。

装完关掉所有终端，重新打开 PowerShell，验证：

```powershell
Get-Command sh
# 期望：sh.exe @ C:\Program Files\Git\usr\bin\sh.exe
```

之后再跑 `claw` 的 bash 工具就正常了。

> ⚠️ 副作用：`C:\Program Files\Git\usr\bin\` 里有一堆与 Windows 同名的 Unix 工具（如 `find.exe`、`sort.exe`），可能会覆盖你预期的 Windows 命令。如果有顾虑，用解法 2。

### 解法 2：只把 `sh.exe` 加到 PATH，不引入其他 Unix 工具

如果已经装过 Git for Windows（任何选项），`sh.exe` 通常已经在 `C:\Program Files\Git\bin\` 或 `C:\Program Files\Git\usr\bin\` 里。手动加到 user PATH：

```powershell
# 找到 sh.exe 实际位置
Get-ChildItem -Path "C:\Program Files\Git" -Recurse -Filter "sh.exe" -ErrorAction SilentlyContinue

# 假设找到的是 C:\Program Files\Git\bin\sh.exe
$existingPath = [Environment]::GetEnvironmentVariable("PATH", "User")
[Environment]::SetEnvironmentVariable("PATH", "$existingPath;C:\Program Files\Git\bin", "User")
```

关闭重开 PowerShell，再次 `Get-Command sh` 验证。

### 解法 3：让 AI 改用 `PowerShell` 工具（不改 PATH）

`claw` 自带一个 `PowerShell` 工具（见 `crates/tools/src/lib.rs:5888 execute_powershell`），它会自动探测 `pwsh` 或 `powershell`，与 sh 完全无关。在你的 prompt 里明示：

```text
> 帮我跑一下 quicksort.py，用 PowerShell 工具，别用 bash
```

或者在 `~/.claw/settings.json` 里把 bash 工具关掉：

```json
{
  "permissions": {
    "deny": ["Bash"]
  }
}
```

然后 AI 会自动改走 `PowerShell` 工具完成等价操作。这种方式优点是不动系统 PATH，缺点是某些 prompt 仍可能写出 bash 风格命令导致 AI 卡住。

### 解法 4：自己在 PowerShell 里跑（最直接）

claw 已经把 `quicksort.py` 写到磁盘上了，文件本身没问题，AI 只是验证失败。直接在 PowerShell 里：

```powershell
python D:\clawtester\quicksort.py
```

这与 claw 是否能跑 bash 无关。代码逻辑用人脑确认 + 手动跑一遍即可。

### 解法 5：给 upstream 提 issue / PR

`crates/runtime/src/bash.rs:269,296` 应该按 hooks 的方式加 `#[cfg(windows)]` 分支：

```rust
#[cfg(windows)]
let mut prepared = {
    let mut p = Command::new("cmd");
    p.arg("/C").arg(command);
    p
};
#[cfg(not(windows))]
let mut prepared = {
    let mut p = Command::new("sh");
    p.arg("-lc").arg(command);
    p
};
prepared.current_dir(cwd);
```

但要注意 `cmd /C` 与 `sh -lc` 不语义等价（quoting、`&&`、变量展开都不同），不是简单替换。更稳的做法可能是探测 `pwsh`/`powershell`/`bash`/`sh` 优先级链。这是 upstream `claw-code` 的一个 Windows 可移植性 task。

## 推荐顺序

1. **先做解法 1（重装 Git for Windows 勾选 Optional Unix tools）** —— 治根，所有 bash 工具调用都好使
2. 如果不想动 Git 安装，做**解法 2**（只把 `sh.exe` 加 PATH）
3. 临时绕开做**解法 4**（PowerShell 里手动跑）
4. 永久绕开做**解法 3**（`deny: ["Bash"]` 让 AI 改用 PowerShell 工具）
